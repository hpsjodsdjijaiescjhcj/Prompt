# TaskForge

[English Version](./README.md)

**Slogan：** Turn vague requests into executable AI tasks.  
**中文副标题：** 把模糊需求锻造成可执行的 AI 任务

TaskForge 是一个面向真实任务场景的 AI 任务澄清与编排产品。  
它不是单纯的提示词工具，也不是只会把用户原话转写成 prompt 的壳层，而是一个把模糊需求逐步转化为可执行任务规范的中间层。

它的核心目标只有一个：**让用户用自然语言提出任务后，系统能够先理解需求、补齐关键信息、对齐任务规范，再根据不同 AI 的能力特点选择更合适的模型，并生成最适配该模型的高质量指令，最终完成交付。**

---

## 产品定位

在大多数 AI 产品中，用户真正遇到的问题并不是“没有模型”，而是：

- 不知道自己该怎么描述任务
- 不知道哪些信息是必须补充的
- 不知道结果为什么不好
- 不知道该选哪个模型
- 不知道如何把一个模糊想法变成一个稳定可执行的请求

TaskForge 解决的不是“模型回答一次问题”，而是**用户需求到 AI 可执行任务之间的断层**。

它的本质是一个任务编排层：

- 向上承接用户自然语言输入
- 向下连接不同模型或未来的 agent 执行器
- 在中间完成澄清、对齐、结构化、模型选择、指令适配与验收

---

## 这个产品真正解决什么问题

### 1. 降低用户表达门槛

很多用户知道自己“想做什么”，但并不知道应该怎样把需求描述成 AI 真正能执行的形式。  
TaskForge 不要求用户一开始就把条件讲全，而是先接住一个模糊请求，再帮助用户补齐关键上下文。

### 2. 把“问一句”升级成“完成一件事”

传统聊天式 AI 更像一次性对话。  
TaskForge 更强调任务闭环：

- 先判断需求是否足够明确
- 如果不明确，就补充关键条件
- 把任务整理成统一规范
- 再进入执行或提示词生成
- 最后还能做结果校验

这意味着它不是只负责“生成内容”，而是负责“提高任务成功率”。

### 3. 减少反复返工

用户体验差的根本原因，很多时候不是模型弱，而是前面缺少必要信息。  
TaskForge 的价值在于：**把返工尽量提前消化在任务澄清阶段，而不是等结果出来以后再反复改。**

### 4. 为未来 agent 自动化打基础

如果未来要把系统继续演进成自动执行、多步骤代理或多模型协同，那么任务的结构化表达就是前提。  
TaskForge 现阶段虽然还不是完整 agent 系统，但它已经在做最关键的前置工作：

- 统一任务表示
- 识别缺失信息
- 对齐目标与验收标准
- 建立可扩展的编排路径

### 5. 不同 AI 要用不同表达方式

同一个任务，交给不同模型时，最优表达方式并不相同。  
有的模型更适合结构化约束，有的模型更适合角色指令，有的模型更擅长代码修改，有的模型更适合长文本分析。

TaskForge 当前阶段一个很现实的价值，就是在任务理解之后继续做两件事：

- 选择更合适的 AI
- 生成更适配该 AI 特点的指令词

这意味着它不只是“帮用户整理需求”，还在帮助用户**把需求翻译成某个具体模型更容易执行好的形式**。

## 核心优势

### 1. 不把用户原话直接丢给模型

这是和很多“提示词包装器”最大的区别。  
TaskForge 不满足于把一句输入机械转写成 prompt，而是会先判断：

- 任务是否足够明确
- 哪些信息已经具备
- 哪些信息缺失但会显著影响结果
- 是否需要先澄清再执行

### 2. 强调“最少必要澄清”

它不是为了多问而多问。  
它的目标是只补充那些真正会影响结果质量的信息，让用户在最少交互成本下得到更稳定的输出。

### 3. 从“任务识别”升级到“任务规范化”

系统不只是做分类，而是把输入逐步整理成统一任务规格。  
这样一来，不同模型、不同执行器、不同任务形态都可以围绕同一份任务规范协作。

### 4. 模型选择与指令适配不是附属能力

很多产品默认把所有模型当成同一种能力接口来使用，这会直接损失效果。  
TaskForge 会把“任务适合什么模型”以及“这个模型应该怎样被指挥”当成核心过程，而不是后处理。

这带来的直接价值是：

- 用户不用自己理解不同模型的差异
- 同一个任务可以得到更贴近模型特性的指令表达
- 模型推荐不只是展示信息，而是直接影响最终交付质量

### 5. 可解释、可演进、可扩展

TaskForge 的流程不是黑盒：

- 为什么要追问
- 还缺哪些关键信息
- 当前任务进入了哪个阶段
- 为什么推荐某类模型或执行方式

这些都可以被清楚展示给用户。  
这让它更适合迭代，而不是停留在“碰运气式出结果”。

### 6. 小模型优先，控制 API 成本

这是这个项目非常重要的一点。  
TaskForge 并不假设所有理解能力都必须依赖昂贵的在线大模型。

项目已经引入了一条轻量小模型路线，用来承担更基础、频率更高的任务：

- 初步路由
- 基础槽位抽取
- 低成本任务理解
- 简单场景下的结构化判断
- 低成本的模型路由前判断

这样做的意义很明确：

- 降低在线 API 调用频率
- 提升系统响应性
- 为后续持续训练和领域优化留出空间
- 让产品更接近可控、可迭代的工程系统，而不是纯粹依赖远程模型的调用器

### 7. 验收层正在从浅层检查升级到“对抗式逻辑验证”

TaskForge 不应该只停留在格式检查或表面结果校验。  
它下一阶段的关键优势，是更强的验证层，核心围绕三件事：

- 用对抗式 verifier 主动寻找逻辑薄弱点
- 用结构化/符号化视角约束任务，而不是只让模型用自然语言互相说服
- 用残差修复方式只修问题局部，而不是整条规划全部重写

这意味着它要把 validation 从：

`结果看起来像是对的`

升级为：

`任务计划在逻辑上是否自洽、在约束下是否可执行、面对边界情况是否会崩`

这也是它从当前编排层走向可靠 agent 系统的桥梁。

## 当前产品思路

TaskForge 现阶段围绕一个非常明确的工作流展开：

`Clarify -> Align -> Execute -> Validate`

### Clarify

系统先判断用户的话是否足够明确。  
如果不够明确，就只追问最关键的问题，而不是抛给用户一整页泛泛表单。

### Align

把用户需求整理成统一任务规范，明确：

- 最终目标是什么
- 作用对象是谁
- 边界条件是什么
- 什么样的结果才算合格

### Execute

根据任务类型、当前信息完整度和可用执行器，选择最合适的执行方式。  
这一步不仅涉及是否执行，也包括：

- 选择更适合当前任务的 AI
- 按模型特点组织更合适的指令格式
- 在现阶段优先产出高质量、可直接使用的模型专属 prompt

也就是说，执行前的“模型选择与指令适配”本身就是产品交付的一部分。

### Validate

结果不是“生成完就结束”。  
系统还会根据任务要求检查输出是否满足目标与约束，这一点对邮件、写作、分析类任务尤其重要。

在当前阶段，验证层已经开始具备第一版原型能力：

- 生成计划草图
- 对抗式发现失败点
- 给出残差修复目标

也就是说，它开始检查的不只是“有没有结果”，还包括“逻辑是在哪一段断掉的”“下一步应优先修哪里”。

## 与普通 Prompt 工具的区别

普通 Prompt 工具的逻辑通常是：

`用户输入 -> 包装成提示词 -> 结束`

TaskForge 的逻辑是：

`用户输入 -> 判断是否明确 -> 补齐关键信息 -> 形成任务规范 -> 选择合适模型 -> 生成适配该模型的指令 -> 路由执行 -> 检查结果`

这意味着它更像一个“任务前处理与编排层”，而不是一个“提示词文本编辑器”。

## 为什么要训练轻量小模型

如果所有任务理解都交给在线大模型，问题很明显：

- 成本高
- 延迟高
- 不够稳定
- 难以做大规模、低成本迭代

所以这个产品不是只做流程设计，也在尝试构建自己的基础理解能力。  
轻量小模型的意义，不是替代所有大模型，而是先承担那些高频、基础、可以被结构化的问题：

- 任务大类判断
- 初步信息抽取
- 缺失槽位识别
- 简单意图识别
- 低成本的模型路由前判断

这样，大模型就只需要在更复杂、更难、更高价值的环节出场。  
这是一条更符合产品长期演进的路线。

随着系统继续发展，这条轻量模型路线还可以进一步扩展到：

- 逻辑漏洞检测
- 边界条件发现
- 局部规划修复
- 昂贵执行前的低成本验证

## 适合的应用场景

TaskForge 适合那些“用户知道自己想做什么，但不一定知道怎么把任务讲清楚”的场景，例如：

- 写作与内容创作
- 邮件与商务表达
- 分析与解释类任务
- 代码与开发类请求
- 需要多条件约束的复杂请求

它尤其适合未来继续向以下方向演进：

- agent 自动执行
- 多模型协作
- 面向垂直领域的任务理解
- 用户长期偏好学习

## 当前阶段的价值

这个项目当前最大的价值，不是“已经替代所有 AI 工具”，而是它已经明确站在一个更正确的方向上：

- 不再把 AI 使用理解成一次性问答
- 不再把 prompt 当成最终产品本身
- 不再只靠堆模型能力解决任务失败
- 开始把“需求理解、任务规范化、模型选择、指令适配、执行编排、结果验收”当成核心产品能力

这也是它未来能够继续走向 agent 自动化的重要前提。

- 帮用户把任务讲清楚

- 帮用户选择更合适的 AI
